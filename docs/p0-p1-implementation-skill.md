# P0/P1 功能实现执行说明

## 使用方式

把下面这段作为 AI 开发任务的入口提示词使用：

> 请先阅读 `AGENTS.md`、`README.md`、`docs/competitor-gap-analysis.md` 和本文档，然后在当前项目中实现 P0 与 P1 功能。实施时必须同时完成前端页面、后端接口、数据表/模型、种子数据和测试，不允许只做静态页面或纯前端假数据。最终功能要可运行、可验收，页面要美观，不能破坏现有登录、阅读器、创作者端、管理端和 RBAC。

## 项目背景

BookRecommendSystem 是 Flask 后端 + Vue 3/Vite 前端的阅读平台原型。当前已有书城推荐、搜索、分类、榜单、书籍详情、在线阅读器、书架/进度、划线/书签/评论、创作者作品/章节/稿件管理、后台审核、评论管理和 RBAC 权限。

本次只实现 `docs/competitor-gap-analysis.md` 中的 P0 和 P1，不做 P2/P3。

## P0：文档与产品表达

必须完成：

- 在 `README.md` 增加竞品差异分析和后续路线图入口，链接到 `docs/competitor-gap-analysis.md`。
- 新增或完善截图索引文档，按读者端、创作者端、管理端分组。
- 新增产品能力地图，展示“发现-阅读-互动-创作-审核-权限”的主链路。
- 明确项目定位：学习型/课程设计/毕设级阅读平台原型，不承诺真实版权、真实收益和真实商业运营。

验收标准：

- 文档为 UTF-8 中文 Markdown。
- 第一次接触项目的人能在 5 分钟内理解项目定位、能力边界和路线图。

## P1：读者端增强

必须完成：

- 听书入口：书籍详情页和阅读器都有入口；可先用浏览器 `SpeechSynthesis API`；包含播放/暂停、上一章/下一章、当前章节标题和不支持提示。
- 阅读统计页：新增 `/user/reading-stats`，展示本周阅读时长、阅读天数、完成章节数、收藏数、划线数、评论数、书签数。
- 书单/书评广场：新增书单或社区入口；支持创建书单、添加书籍、发布书评、列表展示和基础互动。
- 章节/段落评论：阅读器支持按章节查看和发布评论，评论要落库。
- 阅读偏好展示：用户中心或阅读统计页展示主题、字号、行高、边距、划线/评论显示开关；阅读器修改后能同步读取。
- 阅读成就：展示首次加入书架、首次划线、连续阅读、完成章节、本周阅读时长等轻量成就。

验收标准：

- 读者能完成“发现书籍 -> 加入书架 -> 阅读/听书 -> 划线/书签/评论 -> 查看统计/兴趣/成就 -> 创建书单或书评”的闭环。
- 页面有加载、空状态、错误提示，桌面端和移动端不明显错位或溢出。

## P1：推荐与搜索增强

必须完成：

- 推荐理由增强：每本推荐书都展示结合用户阅读、书架、搜索或反馈生成的原因，不能全部是固定文案。
- 用户兴趣标签：从阅读历史、书架、搜索历史、推荐反馈计算，并提供接口和页面展示。
- 榜单筛选：榜单支持 `type`、`period`、`category_id` 等参数，前端有筛选控件。
- 热搜趋势：搜索页展示搜索词、搜索次数、最近搜索时间、趋势状态，来源于真实统计或落库数据。
- 推荐反馈闭环：“不感兴趣”后同一用户不再看到该作品；“更多类似”提升同分类或同标签作品曝光。
- 推荐位配置预留：至少完成推荐位数据表、模型、接口和测试；有余力再做管理端页面。

验收标准：

- 推荐流能解释推荐原因。
- 用户反馈会影响后续推荐结果。
- 搜索页和榜单页使用真实接口数据。

## 数据库要求

- 新增功能必须有合理数据表或字段，不要只写前端假数据。
- 更新 `schema.sql`，必要时更新 `mock_seed_compatible.sql`。
- 表设计要包含必要索引和业务字段，例如 `user_id`、`book_id`、`created_at`、`updated_at`。
- 命名和现有 `app/models.py` 风格保持一致。

建议补充的数据对象：

- 书单、书单图书关联。
- 书评、书评互动。
- 章节/段落评论。
- 阅读统计或阅读会话。
- 用户成就。
- 热搜统计。
- 推荐位配置。

## 前端要求

- 使用 Vue 3 + TypeScript + Element Plus + Tailwind CSS。
- 风格与现有页面一致，页面要真实可用，不做营销页。
- 新增页面要有清晰导航入口。
- 表单、按钮、卡片、筛选、空状态、加载态和错误态要完整。
- 不破坏现有路由守卫、用户 token、管理员 token 和创作者权限逻辑。

## 后端要求

- 使用 Flask + SQLAlchemy，接口风格与现有 `/api`、`/user`、`/admin` 保持一致。
- 登录态接口使用现有 `login_required` / `login_optional`。
- 新增接口要返回清晰 JSON，错误码合理。
- 后台治理相关评论类型要预留字段，方便后续管理端统一处理。

## 测试要求

必须补充或完善测试，至少覆盖：

- 阅读统计接口。
- 书单创建和查询。
- 书评发布和查询。
- 章节/段落评论发布和查询。
- 用户兴趣标签。
- 推荐反馈影响推荐结果。
- 热搜趋势。
- 榜单参数。
- 推荐位配置接口。

最终至少运行：

```powershell
cd frontend
npm run build
```

后端测试按项目实际测试命令运行；如果项目没有测试框架，需要补最小 pytest 配置并运行关键接口测试。

## 最终交付说明

最终回复必须包含：

- 完成的 P0/P1 功能。
- 新增/修改的数据表和接口。
- 新增/修改的页面入口。
- 测试命令和结果。
- 未完成项或后续建议。

实施时不要做无关重构，不要删除用户已有改动。
